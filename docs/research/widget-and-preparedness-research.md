# Stat Smith — Widget & Raid-Preparedness Research Report

Research window: 2026-06-11 · repo branch `development` @ `230d883` · game state: Midnight (12.0.5 live, 12.0.7/Sporefall imminent ~2026-06-16)

This report consolidates a four-phase research program: a complete audit of the existing dashboard and its data sources (Phase 0), a leadership-grounded metrics catalog for the current game (Phase 1), a verified slate of new widget proposals (Phase 2), and cross-source "raid preparedness" composite concepts plus new-source access research (Phase 3). Every external claim carries a URL; every repo claim carries a file/line citation; every data path that was not verified carries an explicit **UNVERIFIED** label (collected in Appendix A.7). No confident guesses appear anywhere in this document.

---

## Executive summary

**What was audited.** All 22 implemented widgets (plus 2 retired, plus the unbuilt backlog), the four integrated data sources (Blizzard Battle.net API, Warcraft Logs v2 GraphQL, Raider.IO REST, the first-party Stat Smith addon → companion → ingest pipeline), and the persistence layer — including the finding that five of six append-only snapshot tables accumulate history that **no code path ever reads back** (`src/server/api/routers/snapshot.ts` reads are latest-only except `snapshot.characterTimeline`).

**What survived verification.** 13 widget candidates entered Phase 2 verification; **10 finalists** emerged (1 merge, 2 drops). Verification was unusually favorable: every Warcraft Logs GraphQL path was live-verified for a measured **19.95 points**, every Blizzard claim was settled in 5 live requests plus offline stored-payload inspection, the one new Raider.IO modifier was live-verified in 1 request, all but one addon Lua API was settled against warcraft.wiki.gg for 12.0.1, and all own-DB claims verified line-exact. **Six finalists are bulletproof** (every data path verified this session, zero or near-zero new external dependencies): W7 engagement_pulse, W1 prog_curve, W3 parse_consistency (core), W13 reset_board, W8 gear_velocity, W9 tier_diary — comfortably above the ≥5 floor.

**Build order (top 3).**
1. **W7 `engagement_pulse`** — churn early-warning from the VaultSnapshot ledger the app writes but never reads; zero new external calls; pathfinds the history-read infrastructure four other finalists reuse.
2. **W1 `prog_curve`** — pull-by-pull progression and night pace; forces the Guild Report Sync (GRS) ingestion into existence, after which W2 and W5 become incremental; ship before Sporefall lands (~5 days after this report).
3. **W3 `parse_consistency`** — median/variance/gap and improvement slopes from rankings data the sync already pays for and discards (grep-verified zero `median*` consumers in `src/`).

**Headline fusion concept (Phase 3).** `tonight_ready` — a raid-night readiness board fusing addon bag-truth (consumables, lockouts, vault claim), Blizzard gear audit, Raider.IO weekly M+, and own-DB freshness gating — is **buildable today** (v1: read-path only, hardcoded thresholds, zero migration). The deepest concept, `learning_curve` (per-player mechanic learning rate from the 12.0 native damage meter's `AvoidableDamageTaken` + WCL deaths + MRT-note assignment exoneration), waits on one bundled addon release and two cheap probes. The standout new source is **Raidbots** (hybrid: public, CORS-open, officially invited file endpoints; no auth; 28-day report expiry); Wipefest is **NO-ACCESS** (no API, scraping ToS-prohibited by the same company whose WCL key this product depends on), WoWAnalyzer is AGPL code with no API (harvest as a link-out target only), WowAudit is a deliberate **skip** (removed from this repo one day before this research, zero organic adoption), and Blizzard's unused namespaces are the cheapest expansion path available (same app token, <1% quota impact).

## 0.2 Integrated sources — used vs unused surface

### Warcraft Logs v2 GraphQL (client-credentials only; `POST https://www.warcraftlogs.com/api/v2/client`)

**Used today** (sole production caller: Tier-A sync stage 6, `tracked-member-sync.ts:783–927`):
- `CHARACTER_ZONE_RANKINGS_QUERY` (`queries.ts:8–25`) — `character.zoneRankings(zoneID, metric: "dps", difficulty: 5)`; only `rankings[].{encounter, rankPercent, bestPercent}` consumed → season `percentile`.
- `buildCharacterEncounterRankingsQuery` (`queries.ts:145–170`) — one aliased `encounterRankings` per boss, **`metric: dps` hard-coded in the query text** (queries.ts:151); only `ranks[].{startTime, rankPercent, report.code, report.startTime}` consumed → `weekPercentile` via local Tuesday-15:00-UTC lockout window.
- `WCL_RAID_ZONES_QUERY` — zone resolution; bypassed in prod by env pin `WCL_RAID_ZONE_ID=46`.
- Dead code: `ZONE_ENCOUNTERS_QUERY` (queries.ts:95–135) is exported and never imported.

**Unused surface (high-value):** the entire `reportData` family (fights, events, tables, masterData, attendance) — nothing in `src/` queries it, though `reportCode` is persisted per weekly best; `guildData.guild.attendance`; non-dps metrics (`hps`, `tankhps`); difficulties other than Mythic 5; the zoneRankings fields already paid for and discarded (`bestPerformanceAverage`, `medianPerformanceAverage`, `medianPercent`, `totalKills`); root `rateLimitData`; user-linked OAuth (deferred to v1.1, `/wcl-callback` is a stub). Phase 2 live-verified most of this surface — see Appendix A.1.

### Blizzard Battle.net API (app token; every endpoint in `profile-{region}` namespace today)

**Used today** (8 sub-resources): OAuth token mint; `/profile/user/wow` (user token, guild discovery only); character `summary`, `/equipment`, `/specializations` (the ONLY source of the talent loadout string on WoW 12.0 — endpoints.ts:53–58), `/mythic-keystone-profile` (+`/season/{id}`), `/encounters/raids`; guild roster `/data/wow/guild/{realm}/{slug}/roster` (profile-namespace gotcha, endpoints.ts:108–110). OIDC login with `checks: ["pkce","state","nonce"]`.

**Unused surface:** zero game-data (static/dynamic) namespaces were touched before this research. High-value unused: achievements **statistics** (crest income, per-boss kill counters — live-verified this session), `/status` (158-byte rename/transfer tombstone), professions, guild activity feed, journal (raid→boss→loot incl. the pre-release 4th-raid instance), M+ season/period index, talent-tree data, item/item-set lookups (first verified use of `static-us` happened in Phase 2 verification — `/data/wow/item/249955`). Full dossier in Phase 3 (§3.2) and Appendix A.2. Notable redundancy: the per-season M+ request contributes nothing but `rawPayload` (~17% of Blizzard Tier-A spend).

### Raider.IO REST (`GET https://raider.io/api/v1/characters/profile`, one call per character per sync)

Fields requested (`tracked-member-sync.ts:216–222`): `mythic_plus_scores_by_season:current`, `mythic_plus_recent_runs`, `mythic_plus_weekly_highest_level_runs`, `raid_progression`, `gear`. **Two of five are dead weight** — `mythic_plus_weekly_highest_level_runs` and `gear` are requested but never read (RIO bills roughly by response size). Used: season score (`scores.all` preferred over Blizzard rating), exact weekly run count ≤10 (recent runs after region reset), season raid progression. Live probes (2026-06-11) verified the full unused field catalog: `talents` (duplicates the Blizzard `/specializations` call's loadout string), `mythic_plus_best_runs` (per-dungeon spread), `:previous` season scores, ladder ranks, plus free default extras (`thumbnail_url`, `last_crawled_at`). `mythic_plus_recent_runs` caps at 10 entries (verified). Raw RIO payloads are never persisted; `SnapshotSource.RAIDERIO` is never written.

### Stat Smith addon → companion → ingest (first-party)

Pipeline: `addon/StatSmith/StatSmith.lua` (v1.1.7, schema 2, Interface 120005) → SavedVariables (`RTS1:<base64>` export, flushed at logout/`/reload`) → companion watcher (~5-min poll) → `POST /uploader/ingest` (bearer token SHA-256 at rest; 512 KB cap, route.ts:28; 40 req/10 min per token) → `AddonUpload` upsert (**one row per character, latest-only** — `characterId @unique`, prisma:557). Three-layer completeness gating prevents partial captures from clobbering good data (data-readiness `complete` flag, never-regress export, server refuse-to-store).

Collected and consumed: vault activities + enum dump (by-name mapping), weekly run count, held keystone, lockouts (H/M), keyword-filtered currencies, consumables buckets, delves, talents (fallback). **Collected but never consumed** (immediate widget fodder, phase0-addon §5): `vault.hasRewards`, `rewardExamples`, per-run `weeklyRuns[].{mapId, level, runDateTime}`, addon-fresh gear links (enchant/gem ids), the entire `inventory` bags/bank block (+`scanned`), `currencies[].{maxQuantity, totalEarned, earnedThisWeek}`, `lockouts[].resetSeconds` + per-boss names + dungeon lockouts, the consumables `other` bucket (healthstones/augment runes), delve `GetHighestRunForCurrentSeason`.

## 1.2 The leadership decision catalog (what metrics must serve)

Five recurring decisions (phase1-leadership research, full citations in that corpus):

| # | Decision | Cadence | Core question |
|---|---|---|---|
| D1 | Bench/swap for a specific boss | Per boss/night | Who are the best 20 for THIS fight? |
| D2 | Coach vs cut an underperformer | Per week | Can this player improve, and how fast? |
| D3 | Strategy change vs execution fix | Per prog session | Is the plan wrong, or the execution? |
| D4 | Burnout/fatigue management | Continuous | Push harder or pull back? |
| D5 | Recruitment bar & trial promotion | Per applicant/trial | Does this player raise the floor? |

Net rule observed in practice: **peak parses gate-keep recruitment conversations; medians, attendance, and mechanics consistency drive bench/cut decisions** (Wowhead WCL guide: median is "a more accurate measure of consistency"; WCL's own recruitment tool filters on median — https://www.warcraftlogs.com/recruitment).

## 1.3 Combat-log metrics (decision-driving unless marked otherwise)

| Metric | Measures | Decision | Consumer | Key confounders |
|---|---|---|---|---|
| **First deaths / death order** (per pull, aggregated) | Who died first/among first N across a pull set | D2 coach (recurring first-death), D1 if persistent, D3 if same *role* dies first | RL, officer, player | Assigned soakers/kiters die "doing their job"; ride-the-wipe deaths; throwaway pulls; melee exposure. "Who died first is almost always the most important death" (https://wowcoach.gg/blog/how-to-read-wow-combat-logs-beginners-guide) |
| **Killing-blow + overkill context** | Cause of death, last 3 hits, last defensive used | D2 (defensive discipline), D3 (unavoidable damage → CD plan) | RL/officer, player | Killing-blow attribution lies (tiny ticking DoTs); overkill disambiguates (https://www.archon.gg/classic-mop/articles/help/death-statistics) |
| **Death timing/clusters + wipe cutoff** | When deaths occur; clustered vs scattered | D3 — clusters = strategy problem, scattered = coaching | RL | Must replicate WCL's wipe cutoff or post-call deaths swamp data |
| **Battle-res economy** | Brez casts/receivers/timing/waste | D3 (reset early-burn pulls), D2, D4 | RL | Charge pool scales with raid size; kill pulls legitimately spend everything. Rarely surfaced by existing tools |
| **Avoidable damage per mechanic** | Hits from dodgeable abilities per player per mechanic | D2 (repeat offenders), D1, D3 (everyone hit → plan wrong) | RL+officer, player | Melee forced overlap; fight length; per-tier curated spell-ID list is "the hard part Wipefest maintains" (https://www.archon.gg/wow/articles/news/wipefest-voidspire-and-dreamrift) |
| **Interrupt/dispel coverage & timing** | Kicks landed per assignment; dispel latency | D3 (rotations), D2 | RL/officer | Raw counts punish correct discipline (only assigned players should kick); some debuffs must ride |
| **Pull-over-pull progression** (best %, phase reached) | Per-pull progress + trend across nights | D4 (call-it-or-push), D3 (plateau at one phase) | RL+officers+team | HP% understates phase bosses (use `lastPhase`); throwaway pulls; comp changes (tooling precedent: https://progstats.io/, WCL MRA — Platinum only) |
| **Consumable/buff-uptime discipline** | Flask/food/pot/rune prep & usage | D2 accountability ("pure effort, zero skill"), D5 screening | Officer, player | Deliberate no-pot on early prog pulls is policy, not laziness; healthstone non-use is *good* if never low |
| **Defensive/major-CD efficiency** | Casts used vs possible + contextual use during damage events | D2 (died with defensive available = most coachable death), D3 (healer CD coverage map) | Player; RL/healing officer | Holding CDs is often correct; fight-length quantization; cross-spec comparison invalid |
| **Parse consistency (median/variance vs best)** | Distribution of percentiles, not the max | D1 ("high median + low variance beats a higher best"), D5, D2 | Officer/RL | The most confounded family: padding, comp, proc luck, selection bias (public logs skew right), gear brackets, **prog parses are "the least accurate and probably most toxic" comparison** (https://us.forums.blizzard.com/en/wow/t/more-reasons-why-parses-should-not-be-taken-seriously/1418515) |
| **Active time / Always Be Casting** | % of fight casting vs idle | D2 — most actionable DPS lever before rotation talk | Player; officer triage | Movement phases, assigned duties, healer idle windows legitimate |
| **Time-of-night fatigue & pace** (pulls/hour, decay by hour) | Pace and quality decay within the night | D4 (call the night), D3 (logistics) | RL+officers only | Mid-night strategy changes; breaks; small samples. **Underserved — no mainstream tool computes fatigue curves** |
| **Attendance/participation per pull** | Pulls each player was in across a prog cycle | D1 fair rotation, D5, D4 | Officer | "Benched" vs "absent" must be distinguished; private/missing logs fake absence |

**Vanity metrics — explicitly NOT decision-driving** (every analytical source warns on these): overall raid-night DPS incl. trash/wipes; best-ever all-ilvl historical parse; raw cross-spec damage-meter rank; total healing/HPS rank (overheal + sniping); raw interrupt/dispel totals; All Star points for roster decisions; **any parse during progression**; killing-blow leaderboards without overkill context. The decision-vs-vanity fork in one line: *death order, avoidable damage trends, medians, priority-target damage, and active time decide; meter toppers, peak parses, and HPS totals decorate.*

## 3.3 Shared design contract for composites (anti-punitive by construction)

Every fusion concept below obeys (from `phase3-fusion.md` §0, post-critique):

1. **Pillar decomposition, never a naked scalar** — labeled chips (pass/warn/fail/**unknown**); roll-ups are counts ("4 of 5 pillars ready"), never opaque weighted indexes.
2. **Every chip clicks to raw evidence** — exact underlying rows with timestamps and deep links (addon `collectedAt`, missing-slot lists, WCL report links, Raidbots link-backs, observation grids).
3. **Unknown ≠ fail** — missing addon upload, stale sync, private logs, unobserved nights, no sim all render grey with a coverage chip, never red. Coverage is structurally uneven; this is the single most important anti-punitive rule.
4. **Contestability is costed shared groundwork — G1 `TeamAnnotation`**: one Prisma model `TeamAnnotation(teamId, characterId, scope, tag, note, authorUserId, createdAt)` + officer-gated tRPC mutations + inline annotate/dismiss UI (~1.5–2 dev-days, owner = first consumer). Backs duty tags (a *correctness* input for C4), excused absences (E1), dismissals (C1/W13), week exclusions (C2), free-text context. Ops note: any prod Prisma migration also requires the rts-worker recreate + next-cache clear.
5. **No cross-spec/cross-role leaderboards** — compare to own history, role-scoped roster median, or an explicit same-spec external reference.
6. **Positive default framing** — default sort is "who needs help / who is owed something"; empty states celebrate; forward-framing language.
7. **Standardized join keys** — `blizzardCharacterId` (rename-proof) and `(region, realmSlug, normalizedName)` via `normalizeKey` (payload.ts:207–208); WCL actor `server` slug-normalized; pull-level joins use `(encounterID, difficultyID, |Δstart| ≤ 60 s)` with per-observer clock-offset calibration + ordinal/duration fallback (addon `startTs` is the raider's local clock); weeks anchor on `weeklyResetBefore()`.

## 3.4 The composite concepts (4 standalone + 1 extension; final slate post-critique)

### C1 — `tonight_ready` · "Raid-Night Readiness Board" — **BUILDABLE TODAY (v1)**

- **Sources fused & unique contribution:** **Addon consumables/lockouts/vault-claim/keystone** (bag truth — zero web API exposes bags; live this-reset lockout state; `hasRewards` currently has zero consumers) + **Blizzard equipment audit** (armory truth: missing slots, tier, ilvl vs the 272–289 Mythic band) + **RIO/Blizzard weekly M+** (`weeklyRunCount` — the only exact ≤10 post-reset count) + **own DB freshness** (`AddonUpload.collectedAt`, `Character.lastSyncedAt` — the gate that makes the rest honest).
- **Fusion logic:** current lockout + latest snapshot per source; five pillar chips per player — Consumables (house-default thresholds: flask ≥ 2, pots ≥ 15, food ≥ 5, weapon-enh ≥ 2 warn-only — **invented for this product**, inspired by but not sourced from the qualitative Method/BoostRoom checklist), Gear hygiene (missing counts = 0; exact slots on warn), Lockout sanity (informational, never pass/fail), Weekly prep (vault claimed, M+ count; cross-links W13), Freshness (raid-night mode greys addon pillars older than ~3 h on raid nights, 24 h otherwise). Team header: "Ready: 14 · Needs attention: 4 · Unknown: 6" + a one-click copyable officer-chat call-out list.
- **Freshness honesty (designed in):** SavedVariables flush only at logout/`/reload` — every addon pillar is labeled "as of last in-game save, Xm ago", never "now"; a documented pre-raid `/reload` ritual is a named workflow cost; an in-addon raid-night nag rides the bundled release (not required for v1).
- **What the composite reveals:** pre-raid, bag-inclusive readiness — bags are addon-only, armory is API-only; no single source or incumbent (wowaudit cannot see bags at all) can render this board.
- **Feasibility: buildable today.** v1 = one tRPC read extension (`buildAddonView` additions for `hasRewards` + `other`-bucket healthstone/augment-rune classification), one widget, hardcoded thresholds, no annotations, zero migration. v1.1 = configurable thresholds (needs the per-widget config UI prerequisite) + G1 dismissals — carried, not hidden.
- **Reliability caveats:** addon coverage gates 3 of 5 pillars (grey unknowns + coverage chip); consumable classification is name-regex (surface the `other` count so misclassification is visible; harden later with Raidbots `flasks.json`/`foods.json`); bank-stored consumables invisible; armory staleness hover.
- **Explainability/contestability:** chip → itemized evidence ("Flask of the Shattered Sun ×3 — saved in-game 2h ago"); v1.1 G1 dismissals ("funneling, ignore gems this week"); raiders see their own row; the board resets each lockout so it can never become a punitive season-long record.
- **Novelty accounting (full disclosure):** grades OVERLAPS-EXISTING on four pillars (re-cuts `consumable_readiness`, `missing_fixes`, `weekly_lockouts`, `vault_progress`/`mplus_weekly`); the net-new material is `hasRewards` consumption, healthstone classification, thresholds, raid-night freshness gating, the cross-source join, and the call-out list. Build plan includes a v2 supersession decision for `consumable_readiness`/`weekly_lockouts` (the `gear_audit`→`missing_fixes` precedent).

### C2 — `practice_payoff` · "Practice Footprint & Payoff Quadrant" — **BUILDABLE TODAY (core); timestamp lane gated**

- **Sources fused & unique contribution:** **own `MplusSnapshot` history** (weekly run-count series — the X-axis backbone, already accruing for everyone) + **RIO** (authoritative score + per-dungeon spread: "one-dungeon farmer" vs broad practice) + **own `VaultSnapshot` + `WclParseSnapshot` histories** (activity breadth + the outcome series — W3-Trend's `rel_w`) + **addon `weeklyRuns[]`** (gated enrichment lane: the only potential source of repeats and run *timestamps* — "ten keys across three off-nights" vs "one carry Sunday 11pm"). The original delve/world lane was **dropped** — no historical world series exists anywhere (`slots.world` is `tracked:false`, addon override is read-time-only); a Blizzard-statistics lane is **deferred** (cumulative counters need an uncosted snapshot-diff table).
- **Fusion logic:** rolling 6 closed raid weeks (min 4); X = practice volume index (keys weighted by level band relative to the player's own rating — weights displayed); Y = W3-Trend's `rel_w` Theil–Sen slope (same qualifying rules; healers/tanks enter only after hps/tankhps ingestion). **Quadrant stability:** MAD-based dead-zones around both medians ("unclassified" grey band) + 2-consecutive-week hysteresis before a label changes. Quadrants: "Grinding & climbing" / "Grinding, plateaued" (coach *method*, not effort) / "Coasting, plateaued" (engagement conversation; cross-link W7) / "Coasting & fine" (explicitly green, to prevent practice-policing).
- **What the composite reveals:** the D2/D5 prerequisite no single source answers — "before we bench X, did X actually try, and did trying work?" RIO alone is effort-without-outcome (the incumbent wowaudit view); WCL alone is outcome-without-effort.
- **Feasibility: core quadrant buildable today** (zero new external calls; both axis series accruing since first sync — no cold start). **Timestamp/repeat lane gated** on a zero-cost stored-payload inspection of `runDateTime` (the collector reads it at StatSmith.lua:283 and the schema passthroughs it, but the field is absent from the documented 12.0 `MythicPlusRunInfo` — if Blizzard renamed it to `completionDate`, the fix is an addon release and the lane's cost class changes). Even when gated open, the fan-out table (`MplusRunObservation`, ~0.5 dev-day) accrues forward-only — the drill-down is partially blind for ~6 weeks post-ship. G1 needed only for week-exclusion annotations (contest layer).
- **Reliability caveats:** correlation ≠ causation, framed as "footprint" with n≈6 weeks displayed; untracked alts hide practice (officer "practices on alt" annotation via G1); carried keys inflate X (level-band weighting + spread mitigate); RIO 10-cap and crawl lag labeled per row.
- **Explainability/contestability:** every placement decomposes to the two axis series; G1 week exclusions render as gaps, not zeros; benched weeks must never read as "coasting" (cross-checked against W5/W6 presence when available).
- **Boundary:** consumes (does not rebuild) W3-Trend's outcome math and would consume W10's substrate if built; W7 owns disengagement alarms — this is the bridge that turns their separate signals into the D2 coach-method/effort/engagement fork.

### C3 — `sim_gap` · "Gear-Limited or Play-Limited? (Sim-Informed Readiness)" — **NEEDS NEW SOURCE: Raidbots (two pre-build gates)**

- **Sources fused & unique contribution:** **Raidbots droptimizer reports** (user-pasted link → fetch `data.json` once → append-only `DroptimizerReport`/`DroptimizerItem`; the ONLY sim-informed signal anywhere in the stack — what an upgrade is *worth* for this exact character; sim freshness is itself a prep behavior) + **Raidbots static data** (`encounter-items.json` boss→loot dictionary; Blizzard journal is the verified fallback) + **WCL** (realized output: v1 = W3's verified overall `zoneRankings` medians; ilvl-bracket normalization is a **gated upgrade** — bracket args are NEEDS-VERIFICATION, nothing ingests them today) + **Blizzard equipment history** (did the upgrade get equipped, and when — closes the loop) + **own DB** (sim-submission freshness per character).
- **Fusion logic:** three pillars — sim freshness (traffic light 14/28 days; 28 = hard expiry; patch-boundary staleness is an officer-set config constant, since Sporefall constants are unverified until 12.0.7), remaining upside (droptimizer total EV as % of sim baseline, banded High/Moderate/Low — never cross-player raw DPS), realized output (overall medians, with the honest weakening stated: overall percentile partially encodes gear → the 2×2 is *advisory* until the bracket gate lands). The 2×2 diagnosis: gear-limited (route loot, don't bench) / play-limited (coaching pack → W2/W3) / converting (feed them) / capped (catalyst/vault choices only). **DPS specs only** — healers/tanks get freshness + loot-wants, no 2×2 (~⅓ of a typical roster outside the headline diagnostic — stated prominently). Plus the loot-council surface: per boss, the roster's stacked wants sorted by EV.
- **What the composite reveals:** sims say what *could* improve; logs say what *did* happen; equipment says what they *wear* — only the join answers "can this player improve with loot, or is loot wasted on them?", and converts loot-council debates from vibes to per-boss EV. No incumbent owns this join either (wowaudit holds wishlists but no parse context; honesty note: this is a view WowAudit *offers* and this product **chose not to adopt** — no demonstrated user demand yet, which matters for prioritization).
- **Feasibility: needs-new-source with an honest, gated cost.** **Gate G-C3b (first, ~$0):** run ONE fresh droptimizer interactively, fetch and **archive its `data.json`** — no live payload has ever been inspected in any research phase (the only evidenced URL 404'd via the 28-day expiry); **the 3–4-dev-day estimate is not credible until this payload exists**; the archive seeds the AGENTS.md-mandated `raidbots-api` skill. **Gate G-C3a:** one WCL introspection+sample probe of bracket args (a few points) before the fair-comparator pillar is claimed. Auth: none; rate limits: none published — compliance = cache locally, link back, fetch each report exactly once, never crawl or auto-submit. Data modeling ~1 dev-day; **parser risk is the real cost** (undocumented `data.json`, SimC profileset name-drift → budget a parser fix per major patch; store raw JSON so reparsing survives expiry). **Workflow cost is bigger than the technical cost:** raiders must run sims and paste links (paste-CTA in their own row; coverage % visible to officers).
- **Reliability caveats:** sims model a stationary fight profile chosen by the submitter (display fight style); spec mismatch vs `CharacterSnapshot.specName` = auto-stale; submission bias (the diligent sim most) → "no sim" is a nudge state, never a red mark.
- **Explainability/contestability:** every EV cell links to the original Raidbots report (the required link-back doubles as the audit trail); 2×2 placement decomposes to the two pillar values with dates; G1 annotations ("sim pending re-gear", "holding for 4pc"); the gear-limited quadrant is styled as a *help* queue, the play-limited quadrant routes to coaching surfaces, never a public flag.
- **Boundary:** W8 shows gearing *pace*, W11 hygiene *latency* — neither knows what gear is *worth*; the dropped W12 was wallet-side, this is value-side; the loot-council surface is net-new territory (no Phase 2 widget touches loot).

### C4 — `learning_curve` · "Mechanic Learning Rate & Repeat-Offense Decay" — **NEEDS ADDON RELEASE**

- **Sources fused & unique contribution:** **Addon native damage meter** (`C_DamageMeter.GetCombatSessionSourceFromType`, `Enum.DamageMeterType` 8 = AvoidableDamageTaken, 9 = Deaths, 5/6 = Interrupts/Dispels; `SecretWhenInCombat` → plain values out of combat — per-pull, per-player avoidable damage **with Blizzard's own avoidability curation**, eliminating the per-tier spell-ID moat; includes every wipe and every unlogged pull; padding-resistant by construction) + **addon roster-at-pull** (`ENCOUNTER_START/END` + `GetRaidRosterInfo` — pull identity + exposure denominators incl. unlogged nights) + **addon MRT note** (`_G.VMRT.Note` — the only machine-readable assignments source anywhere: turns "took avoidable damage" into "did their job") + **WCL GRS deaths/tables layer** (public auditable ground truth, death timestamps = the alive-time normalizer, history pre-dating the addon release; `table(dataType: DamageTaken, abilityID:…)` is **NOT yet live-verified** — gated) + **G1 annotations** (duty tags are a correctness dependency — they change the decay computation).
- **Fusion logic:** pull spine = addon `pulls[]` fanned out to `RaidPull`/`RaidPullAttendee`; WCL fights joined with clock-offset calibration; union with per-pull provenance. Per (player, boss): avoidable-damage rate and death indicator, **exposure-normalized against time-alive** (death-censoring guard: early deaths accrue less avoidable damage — on the addon-only lane, normalize per participation and ALWAYS render the avoidable series paired with the death-rate series). Learning rate = late-bucket ÷ early-bucket rate; **the flag is team-relative by definition**: player-ratio ÷ team-ratio ≥ 1.3 over ≥30 pulls (cancels the progression-depth confounder — later pulls reach deeper phases with new mechanics; everyone-flat = strategy problem, not coaching). Flag floor (≥30 pulls) EQUALS the first-30-pulls amnesty window — no player can be flagged and amnestied simultaneously. MRT-parse + G1 duty tags → "assigned soaker" badge + exclusion from decay.
- **What the composite reveals:** the measurable version of "responsiveness to feedback" — phase 1's D2 deciding signal, today tracked entirely by officer memory. WCL alone misses unlogged nights and censors wipes from rankings; the addon alone has no pre-release history and no per-ability detail; MRT alone is text. Wipefest cannot be integrated and scores against global kills, not within-team learning.
- **Feasibility: needs the bundled addon release** — payload.ts additions `pulls: luaArray(pullSchema)` + `mrtNote: z.unknown()`, all-optional; **payload-size budget corrected:** a ~40-pull columnar ring buffer (~2 raid nights; ≈9k values ≈ 60–120 KB JSON, ~80–160 KB base64) fits the verified 512 KB ingest cap — the original 200-pull design was over-cap; an ack/cursor upload contract (~1–2 dev-days) is named and out of scope. **Gates:** (1) one in-game `/dump` probe (shared with W6's): per-source `C_DamageMeter` reads return plain values out of combat in a raid instance, and combat sessions map 1:1 to attempts and remain readable at `ENCOUNTER_END`/`PLAYER_REGEN_ENABLED`; (2) a ~2-pt WCL probe of `table(dataType: DamageTaken, abilityID:…)` before the degraded WCL-only mode (hand-curated 3–4 mechanics per boss, 1–2 dev-days/tier) is promised. Persistence: `RaidPull`, `RaidPullAttendee`, `RaidPullMeter` append tables; **G1 required**. Coverage: single-runner whole-raid — one diligent officer covers every player.
- **Reliability caveats:** `C_DamageMeter` is new and moving (12.0.1 already changed signatures) — `z.unknown()` + feature-detection; Blizzard's anti-combat-advantage philosophy sanctions post-hoc analytics, but that is an inference, not a guarantee. Avoidable ≠ blameworthy (forced overlaps, strategy soaks, melee exposure) — mitigated by assignment badges, the team-relative flag, and never rendering a flag on <8 pulls. MRT parsing is free-text convention parsing — phase 1 ships raw-note display + manual tags. Cross-lane double counting: addon lane wins totals, WCL lane wins ability detail/death times; provenance always displayed.
- **Explainability/contestability:** a "flat" flag decomposes to the paired per-pull series chart with team-median overlay, provenance, ability names + WCL deep links + a ToS-safe Wipefest link-out; contest paths = G1 duty tags (immediate exoneration + recompute) and "strategy changed on pull N" markers from W1's night clustering. A flag may only render alongside its evidence chart, never in a standalone ranked list; the coaching-candidate list is officer-only.
- **Boundary:** W2 owns death-order snapshots; this owns the *trend over pulls* and the avoidable-damage axis plus the addon lane W2 doesn't have. W1 owns raid-level progression; this is player-level learning. `brez_economy` slots into the same pull spine later.
